- **What was tested:** idempotency for every new job handler (run twice, assert no duplicate side effects) plus a real admin-API idempotency gap found and fixed (`PATCH /admin/orders/:id/assign` was re-sending the dispatch notification on a same-Woshman/partner retry — fixed with an `assignmentChanged` guard, regression test proves both the no-op-retry and genuine-reassignment cases). `npm run typecheck`/`lint`/full test suite green in CI. Manual: a local proxy-based kill/restart check, superseded by the real staging verification above.
- **Deferred / explicitly not done:** rate limiting, Sentry, backups, and alert-threshold configuration — all explicitly scoped to Phase 7, not started here.
- **Issues found during the phase:** two real bugs caught before merge — BullMQ silently rejects any custom `jobId` containing a colon ("Custom Id cannot contain :"), so every `scheduleJob`/`cancelJob` call was failing silently until `jobIds.ts`'s separator was changed from `:` to `-`; and ioredis's default indefinite reconnect retry meant a Redis outage would never actually trip the `try/catch` around scheduling, fixed with a bounded `connectTimeout` plus a `withTimeout()` race. Also: this row itself was left at "Not started" in the phase tracker above after the actual merge — an example of exactly the gap CLAUDE.md's "update on every merge" rule exists to prevent — backfilled here on 2026-07-24 while updating the log for Phase 7.

### Phase 7 — Security & production hardening

- **Merged:** 2026-07-24, PR #20 (https://github.com/zortojnr/woshmart-project/pull/20)
- **CI:** green on merge — build run 30093115180 (merge commit `d14810a`)
- **Exit criteria met:** audit-and-fix pass completed item by item (full mapping was the PR description) — rate limiting active and tested (per-phone/global webhook, per-admin/IP Admin API, outbound throttle); every Admin API route confirmed already Zod-validated; money confirmed integer-kobo everywhere via a full grep sweep; git history (65 commits) swept for secrets, none found; PII-in-logs review found and fixed one real gap (`quote.ts` logging the customer's raw address at error level); error tracking wired end-to-end against a mocked Sentry module (real-DSN verification still pending, tracked below); alert thresholds configured per `CLAUDE.md`'s philosophy, with an explicit test proving a single transient failure does not page. - **What was built:** `src/lib/rateLimiter.ts` (Redis fixed-window counter) plus webhook and Admin API rate-limit middleware; an outbound send throttle in `send.service.ts`; `src/lib/sentry.ts` + a super_admin-only `POST /admin/diagnostics/test-error` route for verification; `src/lib/alertEmail.ts` (a narrowly-scoped urgent email, fired only when a payment/data-integrity-critical job dead-letters); a PII fix in `quote.ts`. A significant follow-on scope, once it was discovered mid-phase that `woshmart-staging-db` is on Render's free tier with a 44-day hard-deletion clock, not just missing backups: `.github/workflows/staging-backup.yml` (daily `pg_dump` → Backblaze B2, 7+ day retention via a B2 lifecycle rule) and `scripts/check-staging-db-deadline.ts` (an escalating reminder email starting well before the deadline, self-updating via the database's own earliest `_prisma_migrations` timestamp rather than a hardcoded date that someone would have to remember to update after a restore).
- **What was tested:** new tests for the rate limiter core (against real Redis), both rate-limit middleware layers, the outbound throttle's spacing behavior, the Sentry-capture wiring, the dead-letter alert-email trigger (and, just as importantly, that it does *not* fire while retries remain), and the deadline-reminder script's edge cases (verified empirically first against real scratch Postgres databases — a brand-new never-migrated database and an empty-but-existing migrations table both fail loudly with a clear, secret-free error rather than crashing or silently computing a wrong date). Full local suite and CI both green on the final merge commit.
- **Deferred / explicitly not done:** production Twilio signature verification (needs real prod console access); actually performing a Render backup/restore test (documented in `docs/SECURITY.md` §3.9, needs Render dashboard access); enabling Render's native alert notifications (documented, needs dashboard access); real end-to-end Sentry capture against a provisioned `SENTRY_DSN`; a data-retention policy decision (flagged as open, not resolved).
- **Issues found during the phase:** three real secret-exposure near-misses happened during this phase's work (Redis password, staging DB password, local dev DB password — all logged in `docs/SECURITY.md` §4.4, all rotated), which surfaced a real gap in how debugging commands were being run rather than a code defect — closed by adding a new non-negotiable rule to `CLAUDE.md` (never print a secret's raw value in tool output/logs/terminal, even for debugging) rather than treating it as a one-off lesson.
